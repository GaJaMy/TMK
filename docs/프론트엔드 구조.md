# TMK 프론트엔드 구조

## 목적

TMK의 웹 레이어를 사용자용과 관리자용으로 분리하고, 각 요구사항 문서를 독립적으로 관리한다.

## 디렉터리 구조

```text
tmk/
├── tmk-api
├── tmk-batch
├── tmk-core
├── tmk-infra
├── tmk-user-web
└── tmk-admin-web
```

## 역할 분리

- `tmk-user-web`: 일반 사용자용 웹 앱
- `tmk-admin-web`: 관리자용 웹 앱
- `tmk-api`: 사용자 API + 관리자 API를 함께 제공하는 단일 Spring Boot 서버
- `tmk-batch`: 시험 자동 제출 배치 서버

## 배포 구조

- `user-web` 컨테이너: 사용자용 웹 앱
- `admin-web` 컨테이너: 관리자용 웹 앱
- `api` 컨테이너: 사용자/관리자 API
- `batch` 컨테이너: 배치
- `caddy` 컨테이너: 외부 요청 라우팅 및 HTTPS

예시 라우팅:

- `https://tmk.example.com` → `tmk-user-web`
- `https://admin.tmk.example.com` → `tmk-admin-web`
- `https://tmk.example.com/api/**` → `tmk-api`
- `https://admin.tmk.example.com/admin/v1/**` → `tmk-api`

## 기존 정적 리소스 처리

- 기존 `tmk-api/src/main/resources/static` 정적 페이지는 더 이상 유지하지 않는다.
- 신규 화면 개발은 `tmk-user-web`, `tmk-admin-web`에서만 진행한다.
- 과거 정적 페이지 구조는 설계 기준으로 사용하지 않는다.

## 요구사항 문서 위치

- 사용자 웹 요구사항: `docs/사용자 웹 요구사항.md`
- 관리자 웹 요구사항: `docs/관리자 웹 요구사항.md`

## 문서 작성 기준

각 웹 요구사항 문서에는 아래 항목을 유지한다.

- 화면 목적
- 라우트 경로
- 필요한 API
- 입력 폼과 검증 규칙
- 상태값과 예외 처리
- 권한 조건
- 완료 기준
