# TMK (Test My Knowledge) 도메인 모델 설계

> 작성일: 2026-03-10
> 버전: v1.0.0
> 아키텍처: 클린 아키텍처 기반 멀티 모듈

---

## 멀티 모듈 구조

```
tmk-parent/
├── tmk-core/               # 도메인 엔티티, 애플리케이션 서비스, outbound port
├── tmk-infra/              # Spring Data JPA repository, persistence adapter
├── tmk-api/                # REST API, 보안, AI/Redis adapter
└── tmk-batch/              # Spring Batch job, scheduler
```

---

## 도메인 계층 구조 (클린 아키텍처)

```
[ Controller / Batch ]
        ↓
[ UseCase / Orchestrator ]              ← tmk-api / tmk-batch
        ↓
[ Application Service / Aggregate ]     ← tmk-core
        ↓
[ Outbound Port ]                       ← tmk-core
        ↓
[ Adapter 구현체 ]                      ← tmk-infra, tmk-api
```

### Port 원칙

- Port는 `DB 기술`이 아니라 `코어가 필요로 하는 동작`을 표현한다.
- 예: `ExamPort.save`, `DocumentPort.findById`, `QuestionGenerationPort.generateQuestions`
- Spring Data JPA 인터페이스는 Port가 아니며, `tmk-infra`에서 Port 구현에 사용된다.
- `tmk-core` 서비스는 repository 대신 Port만 주입받는다.

### Port 패키지 분류

- `com.tmk.core.port.out.persistence`: DB 영속화, 조회, 배치 정리
- `com.tmk.core.port.out.ai`: 임베딩, 텍스트 추출, 문제 생성
- `com.tmk.core.port.out.cache`: 리프레시 토큰, 블랙리스트
- `com.tmk.core.port.out.security`: 비밀번호 인코딩

## 1. User 도메인

### Aggregate Root: User

```
User
├── id: UserId                      (Value Object)
├── email: Email                    (Value Object)
├── password: EncodedPassword       (Value Object, nullable)
├── provider: Provider              (Enum)
├── providerId: String              (nullable)
├── createdAt: OffsetDateTime
└── updatedAt: OffsetDateTime
```

### Value Objects

```
UserId
└── value: Long

Email
└── value: String
    규칙: RFC 5322 이메일 형식 검증

EncodedPassword
└── value: String
    규칙: bcrypt 인코딩된 문자열. 소셜 로그인 사용자는 null 허용

Provider (Enum)
└── LOCAL | GOOGLE | KAKAO | NAVER
```

### 도메인 규칙

- 이메일은 시스템 내 유일해야 한다.
- `provider = LOCAL`인 경우 password는 반드시 존재해야 한다.
- `provider != LOCAL`인 경우 providerId는 반드시 존재해야 한다.
- 회원가입은 이메일 인증이 완료된 후에만 가능하다. 즉, `EmailVerification.verified = true` 상태가 아니면 회원가입 요청을 처리하지 않는다.

### Outbound Port

```java
public interface UserPort {
    User save(User user);
    Optional<User> findById(Long id);
    Optional<User> findByEmail(String email);
    Optional<User> findByProviderAndProviderId(Provider provider, String providerId);
    boolean existsByEmail(String email);
}
```

## 2. EmailVerification 도메인

### Aggregate Root: EmailVerification

```
EmailVerification
├── id: Long
├── email: Email                    (Value Object)
├── code: VerificationCode          (Value Object)
├── verified: boolean
├── expiredAt: OffsetDateTime
└── createdAt: OffsetDateTime
```

### Value Objects

```
VerificationCode
└── value: String (6자리 숫자)
    규칙: 6자리 숫자 문자열
    생성: 랜덤 6자리 숫자
```

### 도메인 규칙

- 인증 코드는 발급 후 5분간 유효하다.
- 인증 코드가 만료된 경우 검증에 실패한다.
- 이미 인증 완료된 코드는 재사용할 수 없다.
- 동일 이메일로 재발급 시 기존 코드는 덮어쓴다.

### Outbound Port

```java
public interface EmailVerificationPort {
    EmailVerification save(EmailVerification emailVerification);
    Optional<EmailVerification> findByEmail(String email);
    void deleteByEmail(String email);
    void deleteExpiredBefore(OffsetDateTime threshold);
}
```

## 3. Document 도메인

### Aggregate Root: Document

```
Document
├── id: DocumentId                      (Value Object)
├── title: String
├── source: String                      (PDF 파일 저장 경로)
├── status: DocumentStatus              (Enum)
└── createdAt: OffsetDateTime
```

### Entity: DocumentChunk (Aggregate 내부 엔티티)

```
DocumentChunk
├── id: Long
├── documentId: DocumentId
├── chunkIndex: int                     (0부터 시작)
├── content: String
└── embedding: float[]                  (1536차원 벡터)
```

### Value Objects / Enums

```
DocumentId
└── value: Long

DocumentStatus (Enum)
└── PROCESSING   파이프라인 실행 중
    COMPLETED    문제 생성까지 완료
    FAILED       처리 실패
```

### 문서 처리 파이프라인

MVP에서는 외부에 공개되지 않은 내부 API를 통해 PDF 문서를 등록하며, 등록 즉시 아래 파이프라인이 실행됩니다. 사용자가 직접 문서를 등록하는 기능은 MVP에 포함되지 않습니다.

```
[내부 API] PDF 수신
    ↓
PDF 텍스트 파싱
    ↓
청킹 (Chunking)         일정 크기로 텍스트 분할
    ↓
OpenAI 임베딩           text-embedding-3-small → 1536차원 벡터
    ↓
pgvector 저장           DOCUMENT_CHUNK 테이블 (PostgreSQL)
    ↓
LLM 문제 생성           유사 청크 검색(ANN) → Prompt 구성 → OpenAI GPT 호출
    ↓
QUESTION 저장           문제 / 선택지 / 정답 / 해설
```

### 도메인 규칙

- 동일한 출처(source)의 문서는 중복 등록할 수 없다.
- 하나의 문서에서 최소 2개 이상의 문제가 생성되어야 한다.
- 문서 등록 직후 `status`는 `PROCESSING`으로 초기화된다.
- 파이프라인이 정상 완료되면 `status`는 `COMPLETED`로 전환된다.
- 파이프라인 중 오류 발생 시 `status`는 `FAILED`로 전환되며, 재처리가 가능하다.

### Outbound Port

```java
public interface DocumentPort {
    Document save(Document document);
    Optional<Document> findById(Long id);
}

public interface DocumentChunkPort {
    List<DocumentChunk> saveAll(List<DocumentChunk> chunks);
    long countByDocumentId(Long documentId);
}
```

## 4. Question 도메인

### Aggregate Root: Question

```
Question
├── id: QuestionId                  (Value Object)
├── documentId: DocumentId          (Value Object)
├── content: String
├── type: QuestionType              (Enum)
├── difficulty: Difficulty          (Enum)
├── answer: String
├── explanation: String
├── options: List<QuestionOption>   (Entity, type = MULTIPLE_CHOICE 일 때만 존재)
├── createdAt: OffsetDateTime
└── updatedAt: OffsetDateTime
```

### Entity: QuestionOption (Aggregate 내부 엔티티)

```
QuestionOption
├── id: Long
├── questionId: QuestionId
├── optionNumber: int               (1~5)
└── content: String
```

### Value Objects / Enums

```
QuestionId
└── value: Long

QuestionType (Enum)
└── MULTIPLE_CHOICE     객관식 (5지선다)
    FILL_IN_BLANK       빈칸 채우기
    IMPLEMENTATION      구현 문제 (소프트웨어 개발 분야)

Difficulty (Enum)
└── EASY    쉬움
    NORMAL  보통
    HARD    어려움
```

### 도메인 규칙

- `type = MULTIPLE_CHOICE`인 경우 options는 반드시 5개여야 한다.
- `type != MULTIPLE_CHOICE`인 경우 options는 비어있어야 한다.
- 문제는 반드시 정답(answer)과 해설(explanation)을 가져야 한다.

### Outbound Port

```java
public interface QuestionPort {
    Question save(Question question);
    Optional<Question> findById(Long id);
    List<Question> findAllByIds(List<Long> ids);
    List<Question> findByTypeAndDifficulty(QuestionType type, Difficulty difficulty);
    List<Question> findByDocumentId(Long documentId);
    Map<Difficulty, List<Question>> findGroupedByDifficulty();
}
```

## 5. Exam 도메인

### Aggregate Root: Exam

```
Exam
├── id: ExamId                              (Value Object)
├── userId: UserId                          (Value Object)
├── examQuestions: List<ExamQuestion>       (Entity)
├── totalQuestions: int
├── timeLimit: int                          (분)
├── status: ExamStatus                      (Enum)
├── startedAt: OffsetDateTime
├── expiredAt: OffsetDateTime
├── submittedAt: OffsetDateTime             (nullable)
└── createdAt: OffsetDateTime
```

### Entity: ExamQuestion (Aggregate 내부 엔티티)

```
ExamQuestion
├── id: Long
├── examId: ExamId
├── questionId: QuestionId
├── orderNum: int
├── myAnswer: String                        (nullable)
└── isCorrect: Boolean                      (nullable, 채점 전)
```

### Value Objects / Enums

```
ExamId
└── value: Long

ExamStatus (Enum)
└── IN_PROGRESS     진행 중
    SUBMITTED       제출됨
```

### 도메인 규칙

- 시험은 최소 10문제로 구성된다.
- 시험에는 각 난이도(EASY, NORMAL, HARD)별 최소 1문제 이상이 포함된다.
- 시험 제한 시간은 기본 30분이며, 문제 수에 따라 증가할 수 있다.
- 시험 시간이 만료(`expiredAt < 현재시각`)된 경우 답안 저장 및 제출이 불가하다.
- 이미 제출(`status = SUBMITTED`)된 시험은 재제출할 수 없다.
- 채점은 제출 시점에 수행되며, 정답률 50% 이상이면 합격이다.
- 합격 기준: `(정답 수 / 총 문제 수) * 100 >= 50`

### 도메인 서비스

#### ExamCreationService

시험 구성 로직을 담당합니다. 각 난이도별 문제를 랜덤으로 선택하여 시험을 구성합니다.

```
ExamCreationService
└── createExam(userId: UserId): Exam
    1. QuestionPort에서 난이도별 문제 조회
    2. EASY / NORMAL / HARD 각 최소 1문제 이상 포함하여 총 10문제 선택
    3. 시험 제한 시간 계산 (기본 30분)
    4. Exam Aggregate 생성 및 반환
```

#### ExamGradingService

제출된 시험의 채점 로직을 담당합니다.

```
ExamGradingService
└── grade(exam: Exam, questions: List<Question>): GradingResult
    1. ExamQuestion의 myAnswer와 Question의 answer 비교
    2. 각 ExamQuestion의 isCorrect 업데이트
    3. 정답 수 집계
    4. 합격 여부 판단 (50% 이상)
    5. GradingResult 반환

GradingResult (Value Object)
├── correctCount: int
├── totalCount: int
├── score: double           (정답률 %)
└── passed: boolean
```

### Outbound Port

```java
public interface ExamPort {
    Exam save(Exam exam);
    Optional<Exam> findByIdAndUserId(Long id, Long userId);
    List<Exam> findByUserIdOrderByCreatedAtDesc(Long userId);
    List<Exam> findExpiredInProgressExams(OffsetDateTime now);
}
```

## 의존성 방향 원칙

```
tmk-api       → tmk-core, tmk-infra
tmk-batch     → tmk-core, tmk-infra
tmk-infra     → tmk-core
tmk-core      → outbound port만 정의
```

- `tmk-core` 서비스는 저장/조회/외부 연동을 Port로만 표현한다.
- JPA repository와 persistence adapter는 `tmk-infra`에 위치한다.
- API 전용 외부 연동(OpenAI, Redis, Security)은 `tmk-api/adapter/out`에 위치한다.
- 도메인 규칙 변경 시 우선 `tmk-core`를 수정하고, 인프라 영향은 adapter 레이어로 제한한다.

## 클린 아키텍처 준수 현황

### 잘 지켜진 부분

| 항목 | 내용 |
|------|------|
| 의존성 방향 | `tmk-api`, `tmk-batch`는 core port를 사용하고, persistence는 `tmk-infra`가 구현 |
| 도메인 순수성 | core 서비스는 repository 대신 port에만 의존 |
| 도메인 로직 위치 | `Exam.isExpired()`, `Exam.submit()`, `ExamQuestion.grade()` 등 Entity 내부에 위치 |
| Port-Adapter 경계 | 인터페이스는 core에, JPA 구현체는 infra에 분리. 의존성 역전 원칙 준수 |
| 예외 처리 | `BusinessException`, `ErrorCode` enum이 core에 위치. API 레이어에서 HTTP 응답으로 변환 |
| 명시적 설정 | API/Batch 애플리케이션이 `tmk-infra.jpa` repository를 스캔 |

### 트레이드오프 (의도적 선택)

| 항목 | 현재 상태 | 이유 |
|------|-----------|------|
| core에 JPA Entity 존재 | 엔티티는 core에 두고 persistence adapter만 infra로 분리 | 완전 분리보다 현재 규모에 더 현실적 |
| api가 core Entity 직접 import | `ExamUseCase` 등이 core Entity 타입을 직접 사용 | 현재 규모에서 허용 범위. 필요 시 command/result 객체로 추상화 가능 |
