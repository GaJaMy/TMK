# TMK (Test My Knowledge) 도메인 모델 설계

> 작성일: 2026-04-27
> 버전: v3.0.0
> 기준 문서: `TMK(Test My Knowledge).md`, `ERD 설계.md`, `API 명세서.md`
> 전제: 현재 구현 코드가 아니라 목표 요구사항 기준 설계

---

## 멀티 모듈 구조

```text
tmk-parent/
├── tmk-core/               # 도메인 엔티티, 애플리케이션 서비스, outbound port
├── tmk-infra/              # JPA repository, persistence adapter
├── tmk-api/                # 단일 Boot Application, 사용자 API + 관리자 API
├── tmk-batch/              # 시험 자동 제출 배치
├── tmk-user-web/           # 사용자 웹
└── tmk-admin-web/          # 관리자 웹
```

- `tmk-core`는 프레임워크에 의존하지 않는 도메인 규칙과 유스케이스를 가진다.
- 관리자 기능은 별도 `tmk-admin` 모듈이 아니라 `tmk-api` 내부 패키지와 `tmk-core`의 관리자 도메인으로 분리한다.
- `tmk-batch`는 만료 시험 자동 제출과 통계 집계 배치를 담당한다.

## 1. UserAccount 도메인

### Aggregate Root: UserAccount

```text
UserAccount
├── id: Long
├── username: String
├── password: String
├── active: boolean
├── countryCode: String
├── createdAt: OffsetDateTime
└── updatedAt: OffsetDateTime
```

### 도메인 규칙

- 일반 사용자는 아이디/비밀번호 기반으로만 가입한다.
- `username`은 일반 사용자 테이블 내에서 유일해야 한다.
- `countryCode`는 개인 문제의 문제/정답/해설 생성 언어를 결정한다.
- 아이디, 비밀번호 외의 개인정보는 저장하지 않는다.
- 로그인이 30일 이상 없는 계정은 자동 삭제 대상이다.
- 비활성 사용자 계정은 로그인, 문서 등록, 시험 생성, 시험 응시에 접근할 수 없다.
- 일반 사용자와 관리자 계정은 별도 저장소를 사용한다.

### Outbound Port

```java
public interface UserAccountPort {
    UserAccount save(UserAccount userAccount);
    Optional<UserAccount> findById(Long id);
    Optional<UserAccount> findByUsername(String username);
    boolean existsByUsername(String username);
    void deleteInactiveUsersBefore(OffsetDateTime threshold);
}
```

### 주요 UseCase

- `RegisterUserUseCase`
- `LoginUserUseCase`
- `ResetPasswordUseCase`
- `LogoutUseCase`
- `ReissueTokenUseCase`

## 2. AdminAccount 도메인

### Aggregate Root: AdminAccount

```text
AdminAccount
├── id: Long
├── username: String
├── password: String
├── active: boolean
├── createdByAdminId: Long?
├── createdAt: OffsetDateTime
└── updatedAt: OffsetDateTime
```

### 도메인 규칙

- 관리자 계정은 기존 활성 관리자만 생성할 수 있다.
- `username`은 관리자 테이블 내에서 유일해야 한다.
- 비활성 관리자 계정은 관리자 로그인 및 관리자 기능 접근이 불가능하다.
- 관리자 삭제는 정책상 논리 삭제 대신 물리 삭제로 단순화하되, 필요 시 활성/비활성으로 운영 가능해야 한다.

### Outbound Port

```java
public interface AdminAccountPort {
    AdminAccount save(AdminAccount adminAccount);
    Optional<AdminAccount> findById(Long id);
    Optional<AdminAccount> findByUsername(String username);
    boolean existsByUsername(String username);
    List<AdminAccount> findAll();
    void deleteById(Long id);
}
```

### 주요 UseCase

- `LoginAdminUseCase`
- `CreateAdminUseCase`
- `GetAdminListUseCase`
- `ChangeAdminStatusUseCase`
- `DeleteAdminUseCase`

## 3. Topic 도메인

### Aggregate Root: Topic

```text
Topic
├── id: Long
├── name: String

├── active: boolean
```

### 도메인 규칙

- Topic은 관리자만 생성/삭제할 수 있다.

- 비활성 Topic은 공용 시험 생성 대상에서 제외된다.

```java
public interface TopicPort {

    boolean existsPublicQuestionByTopicId(Long topicId);
    void deleteById(Long id);
}
```

## 4. Document 도메인

### Aggregate Root: Document

```text
Document
├── id: Long
├── userId: Long
├── title: String
├── sourceType: DocumentSourceType
├── sourceReference: String
├── status: DocumentStatus
├── generatedQuestionCount: int
├── createdAt: OffsetDateTime
└── updatedAt: OffsetDateTime
```

### Entity: DocumentChunk

```text
DocumentChunk
├── id: Long
├── documentId: Long
├── chunkIndex: int
├── content: String
├── embedding: float[]
└── createdAt: OffsetDateTime
```

### 도메인 규칙

- 사용자는 PDF 업로드 또는 `.md` 파일 업로드로 문서를 등록할 수 있다.
- 원본 파일은 문제 생성 완료 후 서버 저장소에서 삭제한다.
- DB에는 문서 메타데이터와 청크/임베딩, 생성된 문제만 남긴다.
- 문서 상태는 `PROCESSING → COMPLETED/FAILED` 흐름을 가진다.
- 개인 문제 생성 개수는 `generatedQuestionCount`로 집계한다.

### Outbound Port

```java
public interface DocumentPort {
    Document save(Document document);
    Optional<Document> findById(Long id);
    List<Document> findByUserId(Long userId);
}

public interface DocumentChunkPort {
    List<DocumentChunk> saveAll(List<DocumentChunk> chunks);
    List<DocumentChunk> findByDocumentId(Long documentId);
    void deleteByDocumentId(Long documentId);
}
```

### 외부 연동 Port

```java
public interface DocumentReaderPort {
    String readPdf(byte[] fileBytes);
    String readMarkdown(byte[] fileBytes);
}

public interface EmbeddingPort {
    List<float[]> embed(List<String> texts);
}

public interface FileStoragePort {
    String storeTempFile(byte[] bytes, String filename);
    void delete(String storagePath);
}
```

### 주요 UseCase

- `UploadDocumentUseCase`
- `GetMyDocumentListUseCase`
- `GetDocumentStatusUseCase`
- `GeneratePrivateQuestionsUseCase`

### Aggregate Root: PrivateQuestion

```text
PrivateQuestion
├── id: Long
├── userId: Long
├── documentId: Long
├── content: String
├── type: QuestionType
├── difficulty: Difficulty
├── answer: String
├── explanation: String
├── languageCode: String
├── options: List<PrivateQuestionOption>
├── createdAt: OffsetDateTime
└── updatedAt: OffsetDateTime
```

### Entity: PrivateQuestionOption

```text
PrivateQuestionOption
├── id: Long
├── privateQuestionId: Long
├── optionNumber: int
└── content: String
```

### 도메인 규칙

- 개인 문제는 사용자 문서를 바탕으로 AI가 생성한다.
- 생성 언어는 사용자 `countryCode`를 기준으로 결정한다.
- `SHORT_ANSWER`는 서술형이 아니라 문서에 존재하는 명확한 단일 정답형만 허용한다.
- `TRUE_FALSE`는 자유 입력이 아니라 2지선다 선택형이어야 한다.
- `MULTIPLE_CHOICE`는 5개 선택지를 가져야 한다.

### Outbound Port

```java
public interface PrivateQuestionPort {
    PrivateQuestion save(PrivateQuestion privateQuestion);
    List<PrivateQuestion> saveAll(List<PrivateQuestion> privateQuestions);
    Optional<PrivateQuestion> findById(Long id);
    List<PrivateQuestion> findByUserIdAndDocumentId(Long userId, Long documentId);
    List<PrivateQuestion> findRandomByDocumentId(Long documentId, int limit);
    long countByDocumentId(Long documentId);
}
```

### 외부 연동 Port

```java
public interface PrivateQuestionGenerationPort {
    List<PrivateQuestionDraft> generate(
        String languageCode,
        String documentTitle,
        List<String> chunks,
        int targetCount

```

## 7. Exam 도메인

### Aggregate Root: Exam

```text
Exam
├── id: Long
├── userId: Long
├── topicId: Long?
├── documentId: Long?
├── sourceType: ExamSourceType
├── totalQuestions: int
├── timeLimitMinutes: int
├── status: ExamStatus
├── startedAt: OffsetDateTime?
├── expiredAt: OffsetDateTime?
├── submittedAt: OffsetDateTime?
├── examQuestions: List<ExamQuestion>
└── createdAt: OffsetDateTime
```

### Entity: ExamQuestion

```text
ExamQuestion
├── id: Long
├── examId: Long
├── publicQuestionId: Long?
├── privateQuestionId: Long?
├── questionScope: QuestionScope
├── orderNum: int
├── myAnswer: String?
└── isCorrect: boolean?
```

### 도메인 규칙

- 시험 생성과 시험 시작은 분리한다.
- 시험 생성 시 상태는 `CREATED`이고, 시작 시 `IN_PROGRESS`가 되며 `startedAt`, `expiredAt`이 설정된다.
- 사용자는 동시에 하나의 `IN_PROGRESS` 시험만 가질 수 있다.
- 사용자가 시험 화면을 이탈해도 `expiredAt`은 변경되지 않으며, 재진입 시 남은 시간은 `expiredAt - now` 기준으로 계산한다.
- `PUBLIC_TOPIC` 시험은 `topicId`를, `PRIVATE_DOCUMENT` 시험은 `documentId`를 반드시 가진다.
- 사용자는 문제 수와 시험 시간(분)을 직접 지정한다.
- 시험 시간 초과 시 사용자의 추가 답안 저장/수정은 불가능하다.
- 시험은 수동 제출 또는 배치 자동 제출로 `SUBMITTED` 상태가 된다.
- `ExamQuestion`은 공용 문제 또는 개인 문제 중 하나만 참조해야 한다.

### Outbound Port

```java
public interface ExamPort {
    Exam save(Exam exam);
    Optional<Exam> findById(Long id);
    List<Exam> findByUserId(Long userId);
    Optional<Exam> findInProgressByUserId(Long userId);
    List<Exam> findExpiredInProgressExams(OffsetDateTime now);
}
```

### 주요 UseCase

- `CreateExamUseCase`
- `StartExamUseCase`
- `GetExamQuestionsUseCase`
- `SaveAnswerUseCase`
- `SubmitExamUseCase`
- `GetExamHistoryUseCase`
- `GetExamHistoryDetailUseCase`
- `AutoSubmitExpiredExamUseCase`

## 8. DailyActivityStat 도메인

### Aggregate Root: DailyActivityStat

```text
DailyActivityStat
├── id: Long
├── statDate: LocalDate
├── userPageAccessAttemptCount: int
├── examRunCount: int
```

- 모니터링 수치는 사용자 웹 접근 시도, 시험 진행, 문서 등록, 개인 문제 생성 4종을 기본으로 한다.

```java
public interface DailyActivityStatPort {

    List<DailyActivityStat> findByStatDateBetween(LocalDate from, LocalDate to);
}
```

### 주요 UseCase

- `GetMonitoringStatsUseCase`
- `AggregateDailyActivityStatUseCase`

## 시험 생성 플로우

### 1. 공용 Topic 시험

```text
사용자 로그인
→ Topic 선택
→ 문제 수 / 시험 시간 입력
→ 활성 공용 문제 랜덤 선택
→ Exam(CREATED) 저장
→ 시작하기
→ Exam(IN_PROGRESS), startedAt/expiredAt 설정
```

### 2. 개인 문서 시험

```text
사용자 로그인
→ 문서 선택
→ 생성 완료된 개인 문제 확인
→ 문제 수 / 시험 시간 입력
→ 개인 문제 랜덤 선택
→ Exam(CREATED) 저장
→ 시작하기
→ Exam(IN_PROGRESS), startedAt/expiredAt 설정
```

## 관리자 운영 플로우

### 1. 관리자 관리

```text
관리자 로그인
→ 관리자 목록 조회
→ 새 관리자 계정 생성

```

### 2. 공용문제 관리

```text
관리자 로그인
→ Topic 생성
→ 공용 문제 등록
→ 문제 상세 조회
→ 문제 활성/비활성 또는 삭제
→ 선택 문제 일괄 활성/비활성 또는 삭제
```

### 3. 모니터링

```text
관리자 로그인
→ 일별/주별/월별 기간 선택
→ 접근 시도 / 시험 진행 / 문서 등록 / 개인 문제 생성 통계 조회
```

## 비고

- 이 문서는 현재 구현 코드가 아니라 요구사항 기준 목표 모델이다.
- `UserAccount`와 `AdminAccount`는 인증 주체가 다르므로 같은 Aggregate로 합치지 않는다.
- `PrivateQuestion`와 `PublicQuestion`도 생성 방식과 관리 정책이 달라 별도 Aggregate로 유지한다.
- 상세 예외 코드와 HTTP 계약은 `API 명세서.md`를 기준으로 본다.
