# 사용자 웹 요구사항

## 문서 목적

`tmk-user-web`에서 구현할 사용자용 화면 요구사항을 정리한다.

## 공통 원칙

- 대상 사용자: 일반 사용자
- 인증 방식: 아이디/비밀번호 + JWT
- 언어 생성 기준: 회원가입 시 저장한 국가 코드
- 문서 원본은 문제 생성 후 서버 저장소에서 삭제
- 랜딩페이지는 비로그인 사용자에게 서비스 소개와 로그인 진입점을 제공해야 한다.
- 아이디, 비밀번호 외의 사용자의 개인정보는 저장하지 않는다.
- 로그인이 30일 이상 없으면 계정은 자동 삭제된다.

## 화면 목록

### 0. 랜딩페이지

- 경로: `/`
- 목적: 서비스 소개, 핵심 기능 안내, 로그인 화면 진입
- 진입 조건: 누구나 접근 가능
- 사용 API: 없음
- 주요 구성:
  - 서비스 소개 Hero 영역
  - 핵심 기능 요약
  - `시작하기` 또는 `로그인` 진입 버튼
- 성공 상태:
  - 사용자가 서비스 목적을 빠르게 이해할 수 있다.
  - 로그인 화면으로 자연스럽게 이동할 수 있다.
- 완료 기준:
  - 비로그인 사용자의 첫 진입 화면 역할을 해야 한다.
  - 로그인 화면으로 이동하는 명확한 트리거가 있어야 한다.

### 1. 인증

정리할 항목:

- 회원가입 화면
- 로그인 화면
- 비밀번호 재설정 화면
- 로그아웃 흐름
- 토큰 재발급 처리 방식

추가 요구사항:

- 로그인 화면에는 `회원가입`, `비밀번호 재설정` 진입 링크가 있어야 한다.
- 비로그인 사용자는 로그인 화면을 중심으로 인증 관련 보조 화면으로 이동할 수 있어야 한다.
- 회원가입 시 `아이디`, `비밀번호`, `국가 코드`를 입력받아야 한다.
- 비밀번호 재설정은 `아이디 입력 및 확인` 단계 이후 `새 비밀번호 입력` 단계로 진행되어야 한다.

기록할 내용:

- 경로
- 입력값
- 유효성 검증
- 성공/실패 UX
- 연동 API

### 2. 홈

- 경로: `/home`
- 목적: 로그인 후 사용하는 사용자 메인 화면
- 진입 조건: 로그인된 사용자
- 사용 API: 없음
- 주요 구성:
  - 오른쪽 상단 로그인한 계정 표시
  - 오른쪽 상단 로그아웃 버튼
  - 왼쪽 네비게이션
  - 문서 등록 진입
  - 시험 보기 진입
  - 시험 내역확인 진입
- 완료 기준:
  - 로그인 성공 후 이 화면으로 이동해야 한다.
  - 현재 로그인한 사용자를 식별할 수 있어야 한다.
  - 왼쪽 네비게이션으로 핵심 기능 진입이 가능해야 한다.

### 3. 문서 등록

- 경로: `/documents`
- 목적: 사용자가 학습 문서를 업로드하고 개인 문제 생성을 시작한다.
- 진입 조건: 로그인된 사용자
- 사용 API:
  - 문서 업로드 API
  - 개인 문제 생성 요청 API
- 주요 입력값:
  - `PDF`
  - `.md`
- 주요 구성:
  - 파일 업로드 입력
  - 지원 형식 안내
  - 업로드 및 문제 생성 실행 버튼
  - 진행 상태 표시
- 진행 상태:
  - `업로드`
  - `문제 생성중`
  - `생성 완료`
- 검증 규칙:
  - 지원하지 않는 형식은 업로드할 수 없어야 한다.
  - 파일 선택 없이 업로드를 시작할 수 없어야 한다.
- 성공 상태:
  - 업로드 이후 문제 생성 흐름이 단계별로 표시되어야 한다.
  - 생성 완료 후 다음 시험 흐름으로 이어질 수 있어야 한다.
- 완료 기준:
  - 문서 등록 페이지가 홈 네비게이션에서 진입 가능해야 한다.
  - 업로드 이후 상태 변화가 사용자에게 명확히 보여야 한다.

### 4. 개인 문제 생성

정리할 항목:

- 생성 상태 표시
- 생성 결과 목록
- 문제 유형별 표현 방식

### 5. 시험

- 경로: `/exams`
- 목적: 진행중인 시험 재진입 여부를 먼저 확인하고, 없으면 새 시험 흐름으로 진입한다.
- 진입 조건: 로그인된 사용자
- 사용 API:
  - 진행중 시험 조회 API
  - 공용문제 시험 생성 API
  - 개인문제 시험 생성 API
- 주요 구성:
  - 현재 진행중인 시험 영역
  - 남은 시험 시간 표시
  - 재진입 버튼
  - 공용문제 시험보기
  - 개인문제 시험보기
  - 별도 시험 진행 화면
- 정책:
  - 시험은 한 번에 하나만 진행할 수 있어야 한다.
  - 시험 도중 이탈해도 시험 시간은 서버 기준으로 계속 감소해야 한다.
  - 진행중인 시험이 있으면 재진입 흐름이 우선되어야 한다.
- 새 시험 흐름 입력값:
  - Topic 또는 개인 문서/문제 묶음 선택
  - 문제 수
  - 시험 시간(분)
- 시험 진행 화면 구성:
  - 좌측 남은 시험 시간 표시
  - 문제 리스트 표시
  - 각 문제별 답안 입력 영역
  - 각 문제별 `확인` 버튼
  - `확인`을 눌러야 답안 입력 완료 상태로 표시
  - 문제 리스트 하단 `최종 제출` 버튼
- 완료 기준:
  - 진행중 시험 여부를 먼저 확인할 수 있어야 한다.
  - 진행중 시험이 없을 때 공용문제/개인문제 두 방식의 시험 진입이 가능해야 한다.

### 6. 시험 내역 확인

- 경로: `/history`
- 목적: 사용자가 이전에 진행한 시험 목록을 확인하고 상세 화면으로 이동한다.
- 진입 조건: 로그인된 사용자
- 사용 API:
  - 시험 이력 목록 조회 API
  - 시험 이력 상세 조회 API
- 주요 구성:
  - 시험 이력 리스트
  - 시험별 불/합 표시
  - 시험 진행 날짜
  - 시험 시간
  - 리스트 항목 클릭 시 상세 화면 이동
- 완료 기준:
  - 리스트에서는 시험 목록만 먼저 보여야 한다.
  - 리스트에서 시험을 선택하면 별도 상세 화면으로 전환되어야 한다.
  - 사용자는 이전 시험의 정오와 해설을 다시 확인할 수 있어야 한다.

### 7. 시험 내역 상세

- 경로: `/history/{examId}` 또는 별도 상세 화면
- 목적: 선택한 시험의 문제별 결과와 해설을 다시 확인한다.
- 진입 조건: 로그인된 사용자, 시험 이력 선택
- 사용 API:
  - 시험 이력 상세 조회 API
- 주요 구성:
  - 문제 목록
  - 문제별 정오 표시
  - 내 답안
  - 정답
  - 해설
  - 목록으로 돌아가기
- 완료 기준:
  - 선택한 시험의 상세 결과가 화면 전환 후 보여야 한다.
  - 사용자는 이전 시험의 정오와 해설을 다시 확인할 수 있어야 한다.

## 비고

- 사용자 웹 요구사항은 기능 요구사항 중심으로 적는다.
- 컴포넌트 분리, 상태관리, 폴더 구조 같은 구현 세부는 별도 설계 문서에서 다룬다.
