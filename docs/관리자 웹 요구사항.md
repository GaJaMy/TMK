# 관리자 웹 요구사항

## 문서 목적

`tmk-admin-web`에서 구현할 관리자용 화면 요구사항을 정리한다.

## 공통 원칙

- 대상 사용자: `ADMIN` 권한 사용자
- 관리자 계정은 기존 관리자만 생성 가능
- 관리자 웹은 공용 문제 운영과 관리자 모니터링 기능 중심으로 제한
- 관리자 웹의 진입점은 관리자 로그인 페이지다.
- 로그인 성공 후 첫 화면은 관리자 메인페이지이며, 메인페이지는 모니터링 화면을 기본으로 한다.
- 관리자 웹은 좌측 네비게이션 영역을 통해 주요 페이지로 이동할 수 있어야 한다.
- 로그인 이후 모든 관리자 페이지의 좌측 네비게이션 상단에는 현재 로그인한 관리자 계정 정보와 로그아웃 버튼이 공통으로 표시되어야 한다.

## 화면 목록

### 관리자 로그인 페이지
- 경로: `/login`
- 목적: 관리자 웹의 진입점으로서 관리자 인증을 처리한다.
- 진입 조건: 비로그인 상태 또는 세션/토큰 만료 상태
- 사용 API: 관리자 로그인 API
- 주요 입력값: 아이디, 비밀번호
- 검증 규칙:
  - 아이디는 필수다.
  - 비밀번호는 필수다.
- 성공 상태:
  - 로그인 성공 시 관리자 메인 화면으로 이동한다.
- 예외 상태:
  - 로그인 실패 시 오류 메시지를 표시한다.
  - 일반 사용자 계정으로 로그인 시 관리자 접근 불가 메시지를 표시한다.
- 완료 기준:
  - 관리자만 로그인 가능해야 한다.
  - 로그인 실패 원인을 사용자에게 이해 가능한 수준으로 안내해야 한다.
  - 로그인 성공 시 관리자 메인페이지로 이동해야 한다.

### 관리자 메인페이지
- 경로: `/`
- 목적: 로그인 후 가장 먼저 진입하는 기본 관리자 화면이며, 운영 모니터링 정보를 중심으로 보여준다.
- 진입 조건: 로그인된 관리자
- 사용 API:
  - 사용자 웹 접근 시도 통계 조회 API
  - 시험 진행 통계 조회 API
  - 사용자 문서 등록 통계 조회 API
  - 사용자 문제 생성 통계 조회 API
- 주요 구성:
  - 좌측 상단 현재 로그인 관리자 정보 영역
  - 좌측 상단 로그아웃 버튼
  - 좌측 네비게이션
  - 상단 페이지 타이틀 또는 현재 위치 표시 영역
  - 중앙 모니터링 대시보드
- 좌측 네비게이션 이동 대상:
  - 관리자 메인페이지(모니터링)
  - 공용문제 관리 페이지
  - 관리자 관리 페이지
- 검증 규칙:
  - 로그인되지 않은 사용자는 접근할 수 없다.
  - 일반 사용자 권한은 접근할 수 없다.
- 성공 상태:
  - 모니터링 핵심 지표와 그래프를 한 화면에서 확인할 수 있다.
  - 현재 로그인한 관리자 계정을 식별할 수 있다.
  - 로그아웃 버튼으로 로그인 화면으로 이동할 수 있다.
  - 좌측 네비게이션을 통해 다른 관리자 페이지로 이동할 수 있다.
- 예외 상태:
  - 통계 데이터가 없으면 빈 상태를 표시한다.
  - 권한이 없으면 접근을 차단한다.
- 완료 기준:
  - 관리자 웹의 기본 진입점 역할을 해야 한다.
  - 좌측 네비게이션이 모든 핵심 운영 페이지 이동 기준이 되어야 한다.
  - 사용자는 현재 어떤 페이지에 있는지 식별할 수 있어야 한다.

### 관리자 관리 페이지
- 경로: `/admins/new`
- 목적: 기존 관리자가 등록된 관리자 목록을 조회하고, 새 관리자 계정을 생성하거나 상태를 관리한다.
- 진입 조건: 로그인된 관리자
- 사용 API:
  - 관리자 목록 조회 API
  - 관리자 계정 생성 API
  - 관리자 계정 비활성화 API
  - 관리자 계정 삭제 API
- 주요 구성:
  - 좌측 상단 현재 로그인 관리자 정보 영역
  - 좌측 상단 로그아웃 버튼
  - 등록된 관리자 목록 표
  - 표 하단 `+` 추가 행
  - 관리자 생성 다이얼로그
- 주요 입력값:
  - 아이디
  - 비밀번호
- 검증 규칙:
  - 아이디는 필수다.
  - 비밀번호는 필수다.
  - 중복 아이디는 허용하지 않는다.
- 성공 상태:
  - 등록된 관리자 목록을 표 형식으로 확인할 수 있다.
  - 표 하단 `+` 행 클릭 시 관리자 생성 다이얼로그가 열려야 한다.
  - 관리자 계정 생성 완료 메시지를 표시한다.
  - 관리자 비활성화 또는 삭제 액션 결과를 표시한다.
- 예외 상태:
  - 권한이 없으면 접근을 차단한다.
  - 중복 아이디면 실패 메시지를 표시한다.
- 완료 기준:
  - 관리자만 접근 가능해야 한다.
  - 등록된 관리자 목록이 표 형식으로 나열되어야 한다.
  - 각 관리자 행에서 비활성화와 삭제 액션을 수행할 수 있어야 한다.
  - 생성 성공 후 즉시 목록에 반영되는 흐름을 가져야 한다.

### 공용문제 관리 페이지
- 경로: `/questions/list`
- 목적: 등록된 공용 문제를 Topic별로 조회하고, 공용 문제 등록 및 Topic 추가를 포함한 관리 작업을 수행한다.
- 진입 조건: 로그인된 관리자
- 사용 API:
  - Topic 목록 조회 API
  - 공용 문제 목록 조회 API
  - 공용 문제 등록 API
  - 공용 문제 활성화 API
  - 공용 문제 비활성화 API
  - 공용 문제 삭제 API
  - Topic 생성 API
- 주요 입력값:
  - Topic 필터
- 주요 구성:
  - 좌측 상단 현재 로그인 관리자 정보 영역
  - 좌측 상단 로그아웃 버튼
  - `문제 관리` 탭
  - `Topic 관리` 탭
  - 문제 관리 탭의 Topic 필터
  - 문제 관리 탭의 문제 목록 표
  - 문제 관리 표 하단 `+` 추가 행
  - Topic 관리 탭의 Topic 목록 표
  - Topic 관리 표 하단 `+` 추가 행
  - Topic 삭제 액션
  - 선택 항목 일괄 활성화/비활성화/삭제 버튼
- 검증 규칙:
  - Topic 미선택 시 전체 또는 기본 Topic 기준 정책을 문서화해 따른다.
- 성공 상태:
  - 선택한 Topic 기준 문제 목록을 표시한다.
  - 공용 문제 등록과 Topic 추가가 같은 관리 흐름 안에서 가능해야 한다.
  - Topic 목록을 별도 탭에서 관리할 수 있어야 한다.
  - Topic 삭제가 가능해야 한다.
  - 각 탭의 표 하단 `+` 행 클릭 시 등록 다이얼로그가 열려야 한다.
  - 문제별 활성화/비활성화/삭제가 가능해야 한다.
  - 선택한 문제들의 일괄 활성화/비활성화/삭제가 가능해야 한다.
- 예외 상태:
  - 권한이 없으면 접근을 차단한다.
- 완료 기준:
  - Topic 기준 조회가 가능해야 한다.
  - 문제의 핵심 정보가 목록에서 식별 가능해야 한다.
  - 개별 관리와 일괄 관리가 모두 가능해야 한다.

### 주간 사용자 모니터링 페이지
- 경로: `/monitoring`
- 목적: 사용자 활동 지표를 기간별로 조회한다.
- 진입 조건: 로그인된 관리자
- 사용 API:
  - 사용자 웹 접근 시도 통계 조회 API
  - 시험 진행 통계 조회 API
  - 사용자 문서 등록 통계 조회 API
  - 사용자 문제 생성 통계 조회 API
- 주요 입력값:
  - 기간 단위 필터(`일별`, `주별`, `월별`)
  - 조회 기간
- 검증 규칙:
  - 기간 단위는 필수다.
  - 조회 기간 형식은 선택한 단위와 일치해야 한다.
- 성공 상태:
  - 아래 지표의 누적 횟수와 그래프를 표시한다.
  - 사용자 웹 페이지 접근 시도 횟수
  - 시험 진행 횟수
  - 사용자 문서 등록 횟수
  - 생성된 사용자 문제 개수
- 예외 상태:
  - 데이터가 없으면 빈 차트/빈 상태를 표시한다.
- 완료 기준:
  - 각 지표를 `일별`, `주별`, `월별`로 전환 조회할 수 있어야 한다.
  - 누적 수치와 그래프가 함께 보여야 한다.
  - 관리자 메인페이지의 기본 콘텐츠로 사용할 수 있어야 한다.

## 비고

- 관리자 웹 요구사항은 운영 기능 범위만 다룬다.
- 사용자 웹과 공통 디자인 요소가 있더라도 요구사항 문서는 분리 유지한다.
- 기존 정적 관리자 페이지는 더 이상 유지하지 않고 신규 관리자 웹을 기준으로 개발한다.
- 좌측 네비게이션은 관리자 웹 전역 공통 UI로 간주한다.
