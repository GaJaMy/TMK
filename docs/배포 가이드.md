# 배포 가이드

## Staging 배포 흐름

- `staging` 브랜치에 push가 발생하면 GitHub Actions가 실행됩니다.
- 워크플로우는 테스트 후 `tmk-api`, `tmk-batch` 이미지를 Docker Hub에 push 합니다.
- 이후 Vultr 서버에 `docker-compose.staging.yml`과 `docker/postgres/init.sql`을 업로드합니다.
- 서버에서 개별 GitHub Secrets를 조합해 `.env.staging` 파일을 생성한 뒤 `docker compose pull`과 `up -d`를 실행합니다.

## Production 배포 흐름

- `main` 브랜치에 push가 발생하면 GitHub Actions가 실행됩니다.
- 워크플로우는 테스트 후 `tmk-api`, `tmk-batch` 이미지를 Docker Hub에 push 합니다.
- 이후 운영 서버에 `docker-compose.prod.yml`과 `docker/postgres/init.sql`을 업로드합니다.
- 서버에서 개별 GitHub Secrets를 조합해 `.env.prod` 파일을 생성한 뒤 `docker compose pull`과 `up -d`를 실행합니다.

## Staging 변수 예시

```env
POSTGRES_DB=tmk
POSTGRES_USER=tmk
POSTGRES_PASSWORD=change-me
POSTGRES_PORT=5432
REDIS_PORT=6379
REDIS_PASSWORD=
API_PORT=8080
DOCUMENT_STORAGE_DIR=/app/uploads
DOCKERHUB_NAMESPACE=your-dockerhub-id
JWT_SECRET=replace-with-a-long-random-secret
OPENAI_API_KEY=
OPENAI_EMBEDDING_MODEL=text-embedding-3-small
OPENAI_CHAT_MODEL=gpt-4o-mini
MAIL_HOST=smtp.gmail.com
MAIL_PORT=587
MAIL_USERNAME=
MAIL_PASSWORD=
SWAGGER_ENABLED=true
BATCH_JOB_ENABLED=false
BATCH_INITIALIZE_SCHEMA=always
```

`IMAGE_TAG`는 워크플로우가 배포 시점의 커밋 SHA로 자동 주입하므로 별도 secret 으로 관리하지 않습니다.

## 서버 준비

- Vultr 인스턴스에 Docker와 Docker Compose Plugin이 설치되어 있어야 합니다.
- `STAGING_DEPLOY_PATH` 예시: `/home/deploy/tmk`
- 최초 1회는 배포 계정이 `docker` 명령을 실행할 수 있어야 합니다.
